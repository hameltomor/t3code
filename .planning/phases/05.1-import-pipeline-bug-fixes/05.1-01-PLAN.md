GAP-2 fix (MEDIUM): In `apps/web/src/components/ImportWizard/steps/SessionListStep.tsx`, replace the `isAlreadyImported` function (lines 23-32) with a version that builds prefixed IDs from the session data before comparing against the `providerThreadIds` set.

The `providerThreadIds` set contains prefixed strings like `"codex:abc123"` (built from `thread.providerThreadId` in `ImportWizard.tsx`). The session summaries contain raw UUIDs in `providerConversationId` and `providerSessionId`. The fix must construct the same prefix format used by `HistoryImportService`: `"${providerName}:${rawId}"`.

Replace with:

```typescript
function isAlreadyImported(
  session: HistoryImportConversationSummary,
  providerThreadIds: Set<string>,
): boolean {
  const prefixedConversation = session.providerConversationId
    ? `${session.providerName}:${session.providerConversationId}`
    : null;
  const prefixedSession = session.providerSessionId
    ? `${session.providerName}:${session.providerSessionId}`
    : null;
  return (
    (prefixedConversation != null && providerThreadIds.has(prefixedConversation)) ||
    (prefixedSession != null && providerThreadIds.has(prefixedSession))
  );
}
```

This approach (Option B from research) is preferred over stripping prefixes because:
- No intermediate Set allocation on every call
- Uses the same `${providerName}:${id}` format documented in HistoryImportService
- `session.providerName` values ("codex", "claudeCode") match the prefix strings exactly
  </action>

  <done>isAlreadyImported correctly matches raw provider UUIDs against prefixed providerThreadId strings by constructing the same prefix format used at import time.</done>
</task>
