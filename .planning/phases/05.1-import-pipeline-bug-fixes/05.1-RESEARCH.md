# Phase 5.1: Import Pipeline Bug Fixes - Research

**Researched:** 2026-03-12
**Domain:** Event-sourced projector / React UI comparison logic / Effect Schema alignment
**Confidence:** HIGH

## Summary

This phase addresses three distinct bugs discovered during the v1.0 milestone audit. All three are well-scoped, localized fixes with clear root causes and minimal blast radius. No new libraries, architectural patterns, or infrastructure are needed -- this is pure bug-fix work against existing code.

GAP-1 (CRITICAL) is a missing field assignment in the in-memory projector that causes `providerThreadId` to always be `null` in the server's read model, breaking both dedup and native resume. GAP-2 (MEDIUM) is a prefix mismatch in the React import wizard that prevents the "Already imported" badge from appearing. The tech debt item is a schema enum that is missing the `"importing"` literal that the materializer writes to the DB.

**Primary recommendation:** Fix all three in a single plan with three tasks. Each fix is 1-5 lines of code change, but each requires a targeted test to prevent regression.

## Architecture Patterns

### Pattern 1: In-Memory Projector Field Propagation

**What:** The projector (`apps/server/src/orchestration/projector.ts`) processes `OrchestrationEvent` objects and builds an `OrchestrationReadModel` in memory. For `thread.created` events, it decodes the payload via `ThreadCreatedPayload` schema and constructs an `OrchestrationThread` object. Fields not explicitly passed to the object literal fall through to `Schema.withDecodingDefault()`.

**Key insight:** The `OrchestrationThread` schema has `providerThreadId` defined as:
```typescript
providerThreadId: Schema.NullOr(TrimmedNonEmptyString).pipe(
  Schema.withDecodingDefault(() => null),
)
```
This means if the field is omitted from the object literal passed to `decodeForEvent(OrchestrationThread, {...})`, the schema silently fills `null`. This is by design for backward compatibility (old events without providerThreadId), but the projector must explicitly pass the field when the payload contains it.

**Current broken code (projector.ts lines 253-276):**
```typescript
const thread: OrchestrationThread = yield* decodeForEvent(
  OrchestrationThread,
  {
    id: payload.threadId,
    projectId: payload.projectId,
    title: payload.title,
    model: payload.model,
    runtimeMode: payload.runtimeMode,
    interactionMode: payload.interactionMode,
    branch: payload.branch,
    worktreePath: payload.worktreePath,
    worktreeEntries: payload.worktreeEntries ?? [],
    latestTurn: null,
    createdAt: payload.createdAt,
    updatedAt: payload.updatedAt,
    deletedAt: null,
    messages: [],
    activities: [],
    checkpoints: [],
    session: null,
    // MISSING: providerThreadId: payload.providerThreadId ?? null,
  },
  event.type,
  "thread",
);
```

**Fix:** Add one line:
```typescript
providerThreadId: payload.providerThreadId ?? null,
```

### Pattern 2: Prefixed Provider Thread ID Comparison

**What:** The `providerThreadId` stored on `OrchestrationThread` uses a prefixed format: `"codex:<uuid>"` or `"claudeCode:<uuid>"`. This prefix is added by `HistoryImportService` at import time (lines 262 and 290). However, the `HistoryImportConversationSummary` from the catalog scanner stores raw UUIDs in `providerConversationId` and `providerSessionId` fields (no prefix).

**Current broken code (SessionListStep.tsx lines 23-32):**
```typescript
function isAlreadyImported(
  session: HistoryImportConversationSummary,
  providerThreadIds: Set<string>,
): boolean {
  return (
    (session.providerConversationId != null &&
      providerThreadIds.has(session.providerConversationId)) ||
    (session.providerSessionId != null &&
      providerThreadIds.has(session.providerSessionId))
  );
}
```

**The set is built in ImportWizard.tsx (lines 60-68):**
```typescript
const providerThreadIds = useMemo(
  () =>
    new Set(
      threads
        .map((t) => t.providerThreadId)
        .filter(Boolean) as string[],
    ),
  [threads],
);
```

This set contains prefixed strings like `"codex:abc123"`, but the comparison checks raw UUIDs like `"abc123"`. They never match.

**Fix options (two viable approaches):**

**Option A (recommended): Strip prefix during comparison.**
```typescript
function isAlreadyImported(
  session: HistoryImportConversationSummary,
  providerThreadIds: Set<string>,
): boolean {
  const rawIds = new Set(
    [...providerThreadIds].map((id) => {
      const colonIndex = id.indexOf(":");
      return colonIndex >= 0 ? id.slice(colonIndex + 1) : id;
    }),
  );
  return (
    (session.providerConversationId != null &&
      rawIds.has(session.providerConversationId)) ||
    (session.providerSessionId != null && rawIds.has(session.providerSessionId))
  );
}
```

**Option B: Build prefixed IDs in the comparison.**
```typescript
function isAlreadyImported(
  session: HistoryImportConversationSummary,
  providerThreadIds: Set<string>,
): boolean {
  const prefixedConversation = session.providerConversationId
    ? `${session.providerName}:${session.providerConversationId}`
    : null;
  const prefixedSession = session.providerSessionId
    ? `${session.providerName}:${session.providerSessionId}`
    : null;
  return (
    (prefixedConversation != null && providerThreadIds.has(prefixedConversation)) ||
    (prefixedSession != null && providerThreadIds.has(prefixedSession))
  );
}
```

**Recommendation: Option B.** It builds prefixed IDs from the session data using `session.providerName` (which is `"codex"` or `"claudeCode"`), matching the exact format used by `HistoryImportService`. This avoids creating a new intermediate Set on every call, and the prefix format is explicitly documented in the codebase. Option A is slightly more resilient to format changes but creates garbage on every call (new Set from spread).

### Pattern 3: Schema Enum Alignment

**What:** The `HistoryImportValidationStatus` schema in contracts defines:
```typescript
export const HistoryImportValidationStatus = Schema.Literals([
  "unknown",
  "valid",
  "missing",
  "stale",
  "invalid",
]);
```

But the `HistoryMaterializer` writes `"importing"` to the DB via `ThreadExternalLinkEntry.validationStatus` (which uses `Schema.String`, not the enum). The `ProvenanceCard` handles `"importing"` in its switch statement (line 32-33), displaying "Partial Import" with a warning badge.

**Fix:** Add `"importing"` to the `Schema.Literals` array:
```typescript
export const HistoryImportValidationStatus = Schema.Literals([
  "unknown",
  "valid",
  "missing",
  "stale",
  "invalid",
  "importing",
]);
```

**Side effects to check:**
- `ThreadExternalLink` in contracts uses `HistoryImportValidationStatus` for its `validationStatus` field -- adding `"importing"` makes this properly typed.
- `HistoryImportValidateLinkResult` also uses the enum -- no impact since validation never returns `"importing"`.
- The `ProvenanceCard` already handles `"importing"` -- this just makes the type checker aware.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| ID prefix format | Custom prefix utility | Inline `${providerName}:${id}` template | The format is simple and only used in two places. A utility would be premature abstraction. |
| Dedup check | Additional DB query fallback | Fix the root cause (projector propagation) | Adding a DB fallback would mask the real bug and add latency |

## Common Pitfalls

### Pitfall 1: Partial Fix of GAP-1 Without Testing Downstream Effects

**What goes wrong:** Fixing the projector but not verifying that `HistoryMaterializer.materialize` dedup and `ProviderCommandReactor.ensureSessionForThread` resume actually work end-to-end.
**Why it happens:** The projector fix is one line, making it seem trivial. But the downstream consumers depend on specific string formats.
**How to avoid:** Write a projector unit test that explicitly asserts `providerThreadId` is non-null after projecting a `thread.created` event with a providerThreadId payload. The existing test at line 41 does NOT assert providerThreadId (it was written before the field existed).
**Warning signs:** Existing test passes without checking providerThreadId -- need to add assertion.

### Pitfall 2: Inconsistent Prefix Format Assumption

**What goes wrong:** Using `codex:` or `claudeCode:` prefix inconsistently between comparison and construction.
**Why it happens:** The prefix is constructed in `HistoryImportService.ts` (lines 262, 290) but consumed in `SessionListStep.tsx`, `HistoryMaterializer.ts` (dedup), and `ProviderCommandReactor.ts` (resume lookup). Different developers might use different prefix logic.
**How to avoid:** The fix for GAP-2 must use the same prefix format as `HistoryImportService` constructs: `"${providerName}:${rawId}"`. Using `session.providerName` from `HistoryImportConversationSummary` matches because the provider name enum values (`"codex"`, `"claudeCode"`) are the same strings used in `HistoryImportService`.
**Warning signs:** Tests pass with hardcoded provider names but break with actual data.

### Pitfall 3: Schema Change Breaking Existing DB Data

**What goes wrong:** Adding `"importing"` to `HistoryImportValidationStatus` could theoretically break if the schema is used for DB reads and existing rows have values not in the enum.
**Why it happens:** Effect Schema `Literals` are strict -- they reject unknown values during decoding.
**How to avoid:** This is not actually a risk here because:
1. `ThreadExternalLinkEntry.validationStatus` uses `Schema.String` (not the enum) for persistence
2. `ThreadExternalLink` in contracts uses `HistoryImportValidationStatus` but the DB already stores `"importing"` -- adding it to the enum makes it decodable, not restrictive
3. The only direction is from restrictive (rejects "importing") to permissive (accepts it)

### Pitfall 4: Existing Test Not Catching the Bug

**What goes wrong:** The existing `projector.test.ts` test "applies thread.created events" passes without `providerThreadId` in expected output because `withDecodingDefault(() => null)` fills it silently.
**How to avoid:** The test must be updated to either (a) include `providerThreadId: null` in the current assertion (to document default behavior) AND add a new test that passes `providerThreadId` in the payload and asserts it appears in the projected thread, or (b) add a separate test case specifically for providerThreadId propagation.
**Recommendation:** Add a new test case rather than modifying the existing one, to keep test intent clear.

## Data Flow Summary

Understanding the full data flow is essential for planning:

```
Import Execute Request (WS)
  -> HistoryImportService.execute()
     -> constructs providerThreadId = "${providerName}:${rawId}"
     -> calls HistoryMaterializer.materialize(input)
        -> checks readModel.threads.find(t => t.providerThreadId === input.providerThreadId)
           ^^^ GAP-1: providerThreadId is always null in memory
        -> dispatches thread.create command with providerThreadId
        -> persists ThreadExternalLink with validationStatus = "importing"
           ^^^ Tech Debt: "importing" not in HistoryImportValidationStatus enum
        -> updates ThreadExternalLink with validationStatus = "valid"

Event Pipeline:
  thread.create command -> decider -> thread.created event
    -> projector.projectEvent()
       -> constructs OrchestrationThread object literal
          ^^^ GAP-1: providerThreadId not included in literal
       -> Schema.withDecodingDefault(() => null) fills it as null

Wizard Session List:
  ImportWizard builds providerThreadIds Set from threads[].providerThreadId
    -> values are like "codex:abc123"
  SessionListStep.isAlreadyImported() compares:
    session.providerConversationId ("abc123") against Set
    ^^^ GAP-2: "abc123" !== "codex:abc123"
```
