## Critical Gaps

### GAP-1: providerThreadId dropped in in-memory projector (CRITICAL)

**File:** `apps/server/src/orchestration/projector.ts` ~line 264
**Root cause:** When projecting `thread.created` events into the in-memory read model, `providerThreadId` from the event payload is not passed to the `OrchestrationThread` object constructor. The schema's `withDecodingDefault(() => null)` fills it as `null`.

**Impact on FR-3 (dedup):** `HistoryMaterializer.materialize` checks `readModel.threads.find(t => t.providerThreadId === input.providerThreadId)` — this never matches because `t.providerThreadId` is always `null` in memory. Re-importing the same session would hit the DB UNIQUE constraint instead of the intended user-friendly rejection.

**Impact on FR-7 (native resume):** `ProviderCommandReactor.ensureSessionForThread` checks `thread.providerThreadId` before doing `ThreadExternalLink` lookup — this is always `null`, so the resume cursor is never constructed. Imported threads start fresh sessions.

**Note:** The persistent DB read model (ProjectionPipeline → ProjectionThreads → ProjectionSnapshotQuery) correctly stores and returns `providerThreadId`. The UI (sidebar badges, source filter, ProvenanceCard) all work correctly because they read from the persistent snapshot. Only the server-side in-memory reactors are affected.

**Fix:** Add `providerThreadId: payload.providerThreadId ?? null` to the object literal in `projector.ts`.

### GAP-2: "Already imported" badge comparison mismatch (MEDIUM)

**File:** `apps/web/src/components/ImportWizard/steps/SessionListStep.tsx` lines 23-32
**Root cause:** `isAlreadyImported` compares raw provider UUIDs (e.g. `"abc123"`) against a set of prefixed `providerThreadId` strings (e.g. `"codex:abc123"`). The comparison never matches.

**Impact:** Sessions that have already been imported never show the "Imported" badge in the wizard's session list. Users could accidentally re-import (though server-side dedup should catch it).

**Fix:** Build the providerThreadIds set with prefix matching, or strip prefix during comparison.

## Tech Debt

### Phase 5: HistoryImportValidationStatus schema gap
- `HistoryImportValidationStatus` enum does not include `"importing"` literal
- Materializer writes `"importing"` to DB via `ThreadExternalLinkEntry` (uses `Schema.String`)
- ProvenanceCard handles it in switch statement at runtime
- Works but is not type-visible on the contract — minor contract/runtime gap
- **Severity:** Low — no functional impact

## Recommendations

1. **Fix GAP-1** (projector.ts): One-line fix, high impact. Unblocks FR-7 native resume and FR-3 proper dedup.
2. **Fix GAP-2** (SessionListStep): Small fix, improves UX by showing "already imported" badges.
3. **Tech debt** (validation status schema): Can be deferred to v1.1 — add `"importing"` to the enum.
